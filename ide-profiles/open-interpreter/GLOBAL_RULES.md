## Mandatory rules

1. **AgentCore authority.** D:\github\agentcore-control-plane is the single source authority. Read order: PROJECT_ANCHOR.md -> DOC_AUTHORITY.md -> CONTEXT_BLOCK.md -> docs/memory-platform/MEMORY_PLATFORM_EXECUTION_PLAN.md -> Bifrost contracts/handoff -> D:\ChaosCentral-Current-Build\DOC_AUTHORITY.md (machine facts). D:\MCP-Control-Plane is evidence only.
2. **One Bifrost gateway entry.** Non-Swarm IDEs use exactly one MCP entry: agentcore-gateway at http://127.0.0.1:8080/mcp with Authorization Bearer ${env:BIFROST_MCP_VIRTUAL_KEY}. Never paste the upstream registry or direct per-tool MCP entries into IDE configs.
3. **Swarm isolation.** SwarmRecall, SwarmVault, and SwarmClaw are a separate ecosystem. Never require Swarm MCP servers, Swarm databases, or Swarm file roots for non-Swarm work. Never modify Swarm product installs. OpenClaw/ClawX are Swarm-managed and outside the non-Swarm baseline.
4. **New Project Bootstrap (Milestone 0).** Every new managed project runs Milestone 0 per docs/agent-policy/NEW_PROJECT_BOOTSTRAP.md before broad implementation: register worktree, load context, preserve the operator prompt verbatim, run Context Fabric capture/drift, discover architecture (Serena/Depwire/Tentra), resolve exact versions via Arabold, create governance files from templates, audit tools.
5. **Effectively-unbounded durable memory.** AgentCore durable project memory is effectively unbounded by model-token limits. Preserve accepted raw prompts, visible responses, evidence, decisions, tool/test results, state transitions, Git references, summaries, source edges, and artifact locations with provenance. Model context limits control only one assembled request. Compaction is non-destructive: summaries are versioned and expandable, and no summary may replace or delete canonical originals. Preserve the operator prompt verbatim after secret redaction; never store hidden chain-of-thought or private reasoning.
6. **Canonical memory recovery before operator repetition.** Never assume information is permanently lost because it is absent from the current chat. Before asking the operator to repeat project history, query agentcore-memory. Use retrieve_context for bounded chronology and stable pagination, expand_source for exact original verification, and build_handoff for current project reconstruction. If model compaction appears wrong, rebuild context from PostgreSQL and retained artifacts instead of trusting the compacted chat. Never directly edit GLOBAL_STATE.md or project STATE.md; they and COMB files are concise generated projections. Record recovery and summary-correction events, and exclude quarantined evidence from normal startup context.
7. **Milestone execution.** Work is organized in Milestones (outcome boundaries) with entry and exit gates per docs/agent-policy/MILESTONE_EXECUTION_STANDARD.md. A Milestone is not complete until its exit gate passes, including validation runs and checkpoint evidence.
8. **Macro and Micro steps.** Macro steps are workstreams with Micro checklists. Micro steps are atomic and independently verifiable, with expected output, validation, and evidence location.
9. **Evidence-backed checklists.** Checklist state is machine-readable (status vocabulary: pending, in_progress, passed, blocked, failed, skipped_with_reason). passed requires an evidence reference. Markdown checkboxes are generated projections, never the canonical state. No step is complete from intention, confidence, or narrative claims.
10. **Context Fabric checkpoints.** Run Context Fabric capture and drift checks at every Milestone entry and exit, and at project bootstrap. Unresolved drift blocks Milestone closure.
11. **Arabold exact-version documentation.** Use arabold-docs (through the gateway) for exact-version dependency and API documentation before dependency or external-API work. Record a documentation checkpoint per Milestone.
12. **Milestone tool audits.** Audit the tool surface at every Milestone entry and exit per docs/agent-policy/TOOL_LIFECYCLE_POLICY.md. Disable expired JIT, completed-Milestone, and unused tools. Keep a tool core_active only with evidence of frequent continued use.
13. **Progressive tool disclosure.** All approved tools remain available for activation, but only tools needed for the current project and Milestone are actively exposed. Projects begin with the Bootstrap profile, never unrestricted administrative or destructive authority. Runtime lease enforcement arrives with memory-platform M6; until then the tool manifest records desired state and agents honor it behaviorally.
14. **Project/worktree write boundaries.** Write only inside the assigned repo/worktree and role-appropriate runtime roots per docs/DRIVE_WRITE_BOUNDARY_RULE.md. Never expose whole-drive filesystem roots. Never format or re-provision H: (live Bifrost runtime).
15. **Secrets policy.** Windows User-scope environment variables only. No .env files. Never print, store, or commit secret values; configs reference env var names only. Missing variable -> stop and report the name.
16. **No direct database access.** Normal IDE agents never connect to PostgreSQL directly and never hold database credentials. Memory flows through agentcore-gateway -> agentcore-memory. Trusted direct SQL exists only in explicitly approved ops/admin runners.
17. **Git safety and validation.** Push after every completed task; narrowest relevant validation plus secret/junk scan first. Never pull/fetch/merge/rebase or force-push without explicit operator instruction. Stage only source-controlled files. See docs/GIT_PUSH_ONLY_POLICY.md.
