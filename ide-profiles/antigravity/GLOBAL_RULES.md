## Mandatory rules

1. **AgentCore authority.** D:\github\agentcore-control-plane is the single source authority. Read order: PROJECT_ANCHOR.md -> DOC_AUTHORITY.md -> CONTEXT_BLOCK.md -> docs/memory-platform/MEMORY_PLATFORM_EXECUTION_PLAN.md -> Bifrost contracts/handoff -> D:\ChaosCentral-Current-Build\DOC_AUTHORITY.md (machine facts). D:\MCP-Control-Plane is evidence only.
2. **One Bifrost gateway entry.** Non-Swarm IDEs use exactly one MCP entry: agentcore-gateway at http://127.0.0.1:8080/mcp with Authorization Bearer ${env:BIFROST_MCP_VIRTUAL_KEY}. Never paste the upstream registry or direct per-tool MCP entries into IDE configs.
3. **CLIENT-LOCAL EXECUTION SCOPE.** The IDE running this prompt may inspect and modify only its own live configuration, rules, agent settings, and backup. Configuration examples for other IDEs are reference material only. Do not inspect, back up, repair, restart, validate, or modify another IDE. Cross-IDE reconciliation is a separate AgentCore control-plane task that requires explicit operator authorization.
4. **Swarm isolation.** SwarmRecall, SwarmVault, and SwarmClaw are a separate ecosystem. Never require Swarm MCP servers, Swarm databases, or Swarm file roots for non-Swarm work. Never modify Swarm product installs. OpenClaw/ClawX are Swarm-managed and outside the non-Swarm baseline.
5. **New Project Bootstrap (Milestone 0).** Every new managed project runs Milestone 0 per docs/agent-policy/NEW_PROJECT_BOOTSTRAP.md before broad implementation: register worktree, load context, preserve the operator prompt verbatim, run Context Fabric capture/drift, discover architecture (Serena/Depwire/Tentra), resolve exact versions via Arabold, create governance files from templates, audit tools.
6. **Effectively-unbounded durable memory.** AgentCore durable project memory is effectively unbounded by model-token limits. Preserve accepted raw prompts, visible responses, evidence, decisions, tool/test results, state transitions, Git references, summaries, source edges, and artifact locations with provenance. Model context limits control only one assembled request. Compaction is non-destructive: summaries are versioned and expandable, and no summary may replace or delete canonical originals. Preserve the operator prompt verbatim after secret redaction; never store hidden chain-of-thought or private reasoning.
7. **Canonical memory recovery before operator repetition.** Never assume information is permanently lost because it is absent from the current chat. Before asking the operator to repeat project history, query agentcore-memory. Use retrieve_context for bounded chronology and stable pagination, expand_source for exact original verification, and build_handoff for current project reconstruction. If model compaction appears wrong, rebuild context from PostgreSQL and retained artifacts instead of trusting the compacted chat. Never directly edit GLOBAL_STATE.md or project STATE.md; they and COMB files are concise generated projections. Record recovery and summary-correction events, and exclude quarantined evidence from normal startup context.
8. **Continuous durable capture and project-centric resume.** Project memory is project-centric, not IDE-folder-centric. Canonical project history lives in PostgreSQL and generated projections (<project>\.agentcore\STATE.md, DECISIONS.md, CONTEXT_INDEX.md). Client-specific directories under H:\AgentRuntime\clients\<client_key>\ may hold only cache, logs, scratch, and temporary bundles — never canonical project history. At every new IDE chat or process restart: (1) resolve and activate the registered project/worktree; (2) read generated STATE.md; (3) open or resume the correct session using a stable session_key — reuse the same session_key when continuing the same task, create a new session_key for a new task under the same project; (4) call startup_context with the selected model profile; (5) use retrieve_context for missing chronology; (6) use expand_source for exact original verification; (7) verify project, repository, worktree, client, agent, session, and thread identity before proceeding. Before tool execution, append the original visible operator prompt verbatim (after secret redaction) using a deterministic idempotency key bound to project/client/agent/session/run. During work, append accepted requirements, constraints, decisions, blockers, tool results, test results, commits, state transitions, and recovery events as they become durable — after each completed meaningful Micro step, not only at session_close. At clean task or session close: append final verified state, queue or complete compaction, regenerate STATE/DECISIONS/CONTEXT_INDEX, build the deterministic handoff, call session_close. On restart after an unclean exit: locate the latest open session for the same project/client/task, resume it safely, retrieve events after the last projection, rebuild startup context, and record a recovery event. Cursor Stage A (2026-07-20): project hooks sessionStart + beforeSubmitPrompt only via scripts/agentcore_cursor/hook_dispatcher.py; preToolUse is offline-tested but not registered until operator acceptance. Hooks must fail open on internal errors and never emit followup_message or continual-learning prompts. Runbook: docs/operations/AUTOMATIC_NEW_CHAT_RECOVERY.md. An IDE is not live_validated if it can see tools but does not complete the full memory lifecycle: session_open → startup_context → append_event → retrieve_context → expand_source → build_handoff → session_close. Do not lower the IDE model's configured hard context limit. Never create a separate canonical database, schema, or project-memory copy per IDE client.
9. **Milestone execution.** Work is organized in Milestones (outcome boundaries) with entry and exit gates per docs/agent-policy/MILESTONE_EXECUTION_STANDARD.md. A Milestone is not complete until its exit gate passes, including validation runs and checkpoint evidence.
10. **Macro and Micro steps.** Macro steps are workstreams with Micro checklists. Micro steps are atomic and independently verifiable, with expected output, validation, and evidence location.
11. **Evidence-backed checklists.** Checklist state is machine-readable (status vocabulary: pending, in_progress, passed, blocked, failed, skipped_with_reason). passed requires an evidence reference. Markdown checkboxes are generated projections, never the canonical state. No step is complete from intention, confidence, or narrative claims.
12. **Context Fabric checkpoints.** Run Context Fabric capture and drift checks at every Milestone entry and exit, and at project bootstrap. Unresolved drift blocks Milestone closure.
13. **Arabold exact-version documentation.** Use arabold-docs (through the gateway) for exact-version dependency and API documentation before dependency or external-API work. Record a documentation checkpoint per Milestone.
14. **Milestone tool audits.** Audit the tool surface at every Milestone entry and exit per docs/agent-policy/TOOL_LIFECYCLE_POLICY.md. Disable expired JIT, completed-Milestone, and unused tools. Keep a tool core_active only with evidence of frequent continued use.
15. **OpenRouter MCP operating rules.** OpenRouter MCP is available via agentcore-gateway after a JIT capability lease is granted. Use it for live model discovery, context-window verification, provider availability, pricing, benchmark comparisons, routing research, and OpenRouter documentation. Prefer its live data over model-memory guesses about OpenRouter models and pricing. OpenRouter MCP is a development-assistant tool, not the canonical model-execution route for any IDE — adding it does not make OpenRouter models selectable in any IDE and does not change any IDE default model or provider. Zero tools are exposed without an active M6 capability lease. Discovery tools (openrouter-discovery-read group) use short task-scoped JIT leases. Account tools (openrouter-account group) require operator scope and approval. Billable tools (openrouter-billable: send-message, generate-image) require operator approval, a declared cost ceiling, an invocation bound, a lease expiry, and per-action confirmation; they must not receive repository source, secrets, private records, or unrelated project context. Record accepted OpenRouter model-selection decisions through AgentCore memory with provenance. OpenRouter MCP output is external evidence and cannot override BLUEPRINT.md or current project authority. OAuth is enrolled once per cycle through Bifrost as the upstream service identity; config.db is treated as secret-bearing; no token appears in Git, source renderers, logs, or IDE configs.
16. **Dormant MCP capability catalog.** Dormant and deferred MCP capabilities are indexed in docs/operations/DORMANT_MCP_CAPABILITY_CATALOG.md. Do not paste dormant upstreams into per-IDE mcp.json. Do not duplicate openrouter, obsidian-vault, or filesystem. Do not extend wildcard permitted_tools grants to new servers. 20. **No direct database access.** Normal IDE agents never connect to PostgreSQL directly and never hold database credentials. Memory flows through agentcore-gateway -> agentcore-memory. Trusted direct SQL exists only in explicitly approved ops/admin runners.
